## 1. 栏目定义与筛选标准

收录目标：各级政府门户网站中**代表本级政府本地新闻/要闻**的稳定栏目列表页（不是单篇文章）。

### 1.1 可接受栏目名称
- `{地名}要闻`（如北京要闻、广州要闻、黄浦要闻）
- 本地要闻 / 区县要闻 / 市县要闻 / 市级要闻 / 政务要闻 / 今日要闻
- 新闻动态 / 工作动态（当上述名称缺失时作为次选）

### 1.2 层级约束（非常重要，极易出错）
- **省级政府**：只收录省级要闻栏目，不要使用下辖地市要闻。
- **市级政府**：只收录市级要闻栏目，**不要使用下辖区县的要闻栏目**，也不要使用"部门动态"栏目替代。
- **区县级政府**：只收录本区县自身的要闻栏目，不使用上级市/省的要闻。

### 1.3 不收录
- 单篇新闻页、专题报道页、转载新闻页
- 短期活动专题、重要讲话数据库等
- 部门动态、部门要闻（通常只反映某一部门工作，不代表本级政府整体）

## 4. 难点类型

1. **首页 JS 渲染**：杭州、成都、厦门、温州等新版门户首页大量依赖前端渲染，`fetch` 拿不到栏目链接，需要 Playwright。
2. **栏目命名多样**：同样是"市级新闻"，可能叫"广州要闻""深圳要闻""要闻动态""政务要闻""今日关注"等，脚本匹配必须使用多关键字集合。
3. **市级与部门动态混淆**：有些站点把"市政府办公厅动态"放在导航首位，容易误收录为市级要闻。必须确认栏目内容是**市级全面新闻**而非单一部门。
4. **统一站群**：某些省份推行市县统一站群（URL 结构类似 `{pinyin}.{province}.gov.cn`），可用于批量发现，但栏目路径不一定一致。
5. **区县要闻命名冲突**：部分区县首页有两个栏目，分别叫"XX要闻"（区政府新闻）和"今日XX"（宣传口/文旅口报道），应采用前者。

## 5. 入口优先级判定

1. 首页主导航栏目名称直接匹配"XX要闻"类关键词。
2. 其次是首页"新闻动态""工作动态""政务要闻"等大版块入口。
3. 再次是站点地图中的"新闻"类一级分类。
4. 如只有单篇标题可见、未找到列表页，放弃并标记为"待复核"，不强行填入单篇链接。

## 6. 查找方法

- 静态优先：`fetch` 门户首页，grep 匹配 "要闻/新闻动态/今日XX/政务要闻" 类链接。
- 站点地图：尝试 `/zt/wzdt/`、`/wzdt/`、`/sitemap.html` 等常见路径。
- 站内搜索：关键词 "要闻"、"本地要闻"、"新闻动态"。
- JS 渲染兜底：使用 Playwright 加载首页后再提取。
- 批量方法仅在已确认同省站群结构一致时使用，并必须抽样人工复核。

## 7. 最终采用原则

- 一级导航可直达 > 站点地图可直达 > 站内搜索匹配。
- 可检索、可按时间倒序浏览的列表页 > 仅显示滚动条样式的 widget。
- 长期稳定 URL > 包含 session/token 参数的短期链接。
- 若一地同时存在多个要闻栏目（市委要闻 / 市政府要闻 / 今日 XX 等），优先政府门户下的"市级要闻"，并在本文件"候选备忘"中保留其他未采用链接及取舍说明。

## 9. 后续扩展建议

1. **省会批量**：先把 31 个省级 + 27 个省会要闻栏目人工核验完，填入数据；这一层一般不吃力。
2. **副省级市 + 计划单列市**：共 15 个（含广州/深圳/青岛/厦门等），单列核验。
3. **地级市群**：以省为单位，分批处理。优先考察同省站群是否存在统一结构（如浙江统一 CMS `col/colXXX/index.html`），若是则可半自动。
4. **区县级**：规模大、结构差异大，需最谨慎。建议按"省会所在市的辖区 → 副省级市辖区 → 其他地级市辖区 → 县/县级市"顺序推进，每个地区手动复核。
5. 脚本草稿：可复用 `scripts/website_management/` 下的结构，添加 `news-site-probe.mjs` 抓取首页并正则匹配要闻栏目候选。
6. 验证：每次批量后抽样 5% 人工点击，更新本日志。
